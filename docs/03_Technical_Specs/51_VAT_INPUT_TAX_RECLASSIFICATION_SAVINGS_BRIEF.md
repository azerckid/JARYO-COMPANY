# VAT Input Tax Reclassification Savings Brief
> Created: 2026-07-12
> Last Updated: 2026-07-12
> Backlog: JC-041
> Status: docs-only pre-code brief; UI-First Gate and owner Preview approval still pending before implementation

## 0. Decision

기장 단계에서 매입 거래가 법정 불공제 사유(현재 범위: 접대비/기업업무추진비)로
분류되면, 그 이후 부가세 판단은 그 분류를 그대로 따라간다. 그러나 분류 자체가
기장 실수였을 수 있다 — 같은 "식대" 지출이라도 참석자가 전부 내부 직원이면
복리후생비·회의비이고, 그 경우 매입세액을 공제받을 수 있다. 지금은 이 가능성을
아무도 다시 확인하지 않는다.

JC-041은 **이미 불공제로 분류된 매입 중, 재분류하면 사업자가 실제로 세금을
아낄 수 있는 거래를 찾아 제안**한다. 목표는 "판단이 맞는지 확인해달라"가 아니라
"**놓치고 있는 절세 기회를 찾아서 구체적인 금액으로 보여준다**"는 것이다.

이 브리프는 [JC-039 Evidence-Backed Decisive Judgment Brief](./50_VAT_AI_EVIDENCE_BACKED_DECISIVE_JUDGMENT_BRIEF.md)의
근거 우선 원칙을 그대로 재사용한다. **근거 없이 재분류를 제안하지 않는다** —
부당공제로 확정되면 가산세 리스크가 있으므로, "혹시 몰라서" 제안하는 것은
사용자에게 손해를 끼칠 수 있다.

## 1. Origin and Motivation

2026-07-12 VUI-1b(부가세 화면 단순화) Preview 리뷰 세션 중 프로젝트 오너가
"확인 필요 거래" 표에 남아있는 불공제 판단(접대비)이 사실은 이미 확정된
사실이라 재확인이 불필요하다는 점을 지적했다. 이어서 "공제 못 받는 것으로
되어 있는 것 중 실제로는 공제받을 수 있는 걸 찾아주면 사업자에게 이득이
되지 않겠냐"는 방향 제안이 나왔고, 이 문서로 정리한다.

핵심 통찰: 부가세법상 공제/불공제는 법이 이미 정한 폐쇄형 분류 체계(제38·39조)라
"AI가 몰라서 물어보는" 영역이 아니다. 반면 **그 거래가 애초에 올바른 카테고리에
분류됐는지는 별개 문제**이며, 여기에는 기장 단계의 실수가 개입할 여지가 있다.
이 실수를 찾아 고치면 사업자에게 직접적인 금전적 이득이 된다.

## 2. Scope

### 2.1 In Scope (1차)

- 부가세 매입세액 불공제 사유 중 **④ 기업업무추진비(접대비) 및 이와 유사한
  비용 관련**으로 분류된 거래만 다룬다.
- 재분류 후보: 복리후생비, 회의비 — 둘 다 참석자가 내부 직원으로 한정될 때
  성립하는 카테고리이며, 부가세 매입세액 공제가 가능하다.
- AI는 기존 JC-039 evidence resolver가 참조하는 소스(거래 적요, 연결 증빙,
  같은 tenant의 과거 확정 이력)에서 재분류를 뒷받침하는 근거를 찾는다.
- 근거를 찾으면 **구체적 절세 금액과 함께** 재분류를 제안한다. 사용자가
  명시적으로 확정해야 실제 분류가 바뀐다.

### 2.2 Out of Scope (1차)

- 접대비 외 다른 법정 불공제 사유(③ 비영업용 소형승용차, ⑤ 면세사업 관련
  등)의 재분류 탐지는 후속 논의. 사유별로 근거 자료 성격이 크게 달라 이번
  범위에 포함하지 않는다.
- 부가세 외 세목(소득세·법인세 등)의 절세 가능성 탐지는 범위 밖.
- 재분류 자동 확정. 기장 계정과목을 AI가 직접 수정하지 않는다.
- 원래 불공제 분류 자체가 틀렸는지(즉 애초에 접대비가 아닌데 접대비로
  기장됐는지)를 포괄적으로 재검증하는 감사 기능. 이건 JC-039의 일반 판단
  범위이며, JC-041은 "불공제 판정을 받아들이되 재분류 여지만 본다."

## 3. Why This Is Different From "확인 필요" (JC-039 대비)

JC-039의 "확인 필요"는 **분류 자체가 불확실**할 때 쓴다(예: 이 매출이
영세율인지 판단할 근거가 부족함). JC-041은 분류가 이미 확정된 상태에서
**"혹시 이 확정이 원래 잘못된 게 아니었나"**를 다시 보는 것이다. 이 둘은
화면에서 섞이면 안 된다 — 불공제 chip을 클릭했을 때 "이미 확정된 사실
설명"과 "혹시 재분류하면 이득" 제안이 뒤섞이면, 사용자가 이미 끝난 판단을
다시 의심해야 하는 것처럼 보인다. UI Contract(§6)에서 이를 분리한다.

## 4. Reclassification Evidence Rules

### 4.1 Positive Evidence (재분류 제안 가능)

다음 중 하나 이상이 확인되면 재분류 후보로 제안한다.

- 거래 적요·메모에 참석자가 명시되어 있고 전부 내부 직원(같은 tenant의
  `employee_profile` 명단과 대조 가능)인 경우
- 적요에 "회식", "워크샵", "전 직원", "팀 미팅" 등 내부 행사를 특정하는
  표현이 있고, 외부 거래처를 특정하는 표현이 없는 경우
- 같은 tenant에서 동일하거나 유사한 패턴(같은 가맹점·비슷한 금액대·정기적
  주기)의 거래를 사용자가 과거에 복리후생비·회의비로 확정한 이력이 있는 경우

### 4.2 Negative Evidence (재분류 제안 금지, 원래 분류 유지)

다음 중 하나라도 있으면 제안하지 않는다.

- 적요에 외부 거래처명·직함(예: "○○사 대표", "거래처") 등이 특정되어 있음
- 참석자 정보가 아예 없어 내부/외부 여부를 판단할 근거가 없음(근거 부족은
  "재분류 가능성 없음"이 아니라 "판단 불가" — 이 경우도 제안하지 않는다.
  JC-039 §3 "근거 없음 -> 특례 해당 없음"과 동일하게, 판단 불가는 원래
  분류를 그대로 둔다)
- 금액이 반복적인 소액 식대 패턴과 다르게 큰 경우(접대성 지출 특징과 일치)
- 사용자가 과거 같은 tenant·같은 패턴 거래를 접대비로 명시적으로 재확정한
  이력이 있는 경우(사용자의 기존 결정 존중, JC-039 §2 확정 행 보호와 동일)

### 4.3 No Evidence Means No Proposal

증거가 애매하거나 부족하면 재분류를 제안하지 않고 원래 분류(불공제)를
그대로 유지한다. "혹시 될 수도 있으니" 식의 낮은 확신 제안은 금지한다 —
사용자가 이를 근거로 재분류를 확정했다가 세무조사에서 부인되면 가산세를
물 수 있고, 이는 이 기능의 목적(절세)과 정반대 결과다.

## 5. User Confirmation Flow

1. AI가 근거를 찾으면 "재분류 후보"로 표시하고, 구체적 절세 금액을 함께
   보여준다.
2. 사용자가 근거를 확인하고 "복리후생비로 재분류"(공제 전환) 또는
   "접대비 유지"(원래 분류 유지)를 명시적으로 선택한다.
3. 재분류 확정 전까지 canonical VAT fact·매입세액공제 계산은 변경되지
   않는다.
4. "접대비 유지"를 선택하면 같은 거래·같은 패턴에 대해 반복해서 다시
   묻지 않는다(사용자 결정을 감사 이력으로 저장, §4.2 negative evidence로
   재사용).

## 6. Tax Savings Framing (제안 문구 요건)

제안에는 반드시 다음이 함께 있어야 한다.

- 현재 분류와 재분류 후 분류(예: "접대비 → 복리후생비")
- 재분류를 뒷받침하는 근거 요약(§4.1 중 어떤 근거가 발견됐는지)
- **구체적 절세 금액**(예: "매입세액 132,000원을 추가로 공제받을 수 있습니다")
- 재분류가 최종이 아니며 사용자 확인이 필요하다는 명시

금액 없이 "재분류 가능성이 있습니다"라고만 하는 것은 이 기능의 핵심 가치
(구체적 이득 제시)를 놓치는 것이므로 허용하지 않는다.

## 7. Data Contract (draft)

기존 `vat_deduction_review`(사용자 확정 감사 테이블)는 최종 분류 결정을
저장하는 곳이다. 재분류 제안은 그 결정에 도달하기 전 단계의 추천이므로,
JC-039의 `evidenceTrace` 구조를 재사용하는 additive 필드로 설계한다
(신규 테이블 여부는 VAI/JC-039 스키마 확정 이후 결정).

| 필드 | 목적 |
|:---|:---|
| reclassificationCandidate | 재분류 후보 여부(boolean) |
| currentCategory / suggestedCategory | 현재 분류(접대비) / 제안 분류(복리후생비·회의비) |
| evidenceFound | §4.1 근거 목록(참석자 대조, 적요 패턴, 과거 확정 이력 등) |
| potentialSavingsKrw | 재분류 시 추가로 공제 가능한 매입세액 |
| userDecision | `pending` / `reclassified` / `kept_as_is` |
| decisionRowId | 사용자 결정을 저장하는 감사 row 참조(재질문 방지용) |

정본은 여전히 `vat_deduction_review`의 사용자 확정값이며, 이 필드들은
추천·이력 계층으로 canonical 값을 대체하지 않는다.

## 8. UI Contract (방향성, Preview 승인 전)

- 재분류 제안은 "확인 필요 거래" 표(JC-039/VUI-1 목표 UI)와 **다른 자리**에
  둔다. 불공제 chip을 클릭했을 때 나오는 "무슨 뜻인가요 / 홈택스에서 할 일"
  팝오버에 재분류 제안을 섞지 않는다 — 이미 확정된 사실 설명과 "다시
  생각해볼 여지"가 같은 화면 요소에 있으면 혼란스럽다.
- 후보안: 절세 후보가 있을 때만 노출되는 별도 배지/섹션(예: "절세 가능
  거래 N건 · 최대 OOO원") — 없으면 아예 표시하지 않는다(예외 0건이면
  숨기는 기존 원칙과 동일).
- 정확한 배치·문구·상태는 실제 화면 논의와 HTML Preview 확인 후 확정한다
  (docs-plan UI-First Gate 준수 — 이 브리프는 코드 작성 전 계약 문서이며
  화면 자체를 확정하지 않는다).

## 10. Acceptance Criteria

- [ ] 근거(§4.1) 없이는 재분류를 제안하지 않는다.
- [ ] 부정 근거(§4.2)가 있으면 제안하지 않고 원래 분류를 유지한다.
- [ ] 모든 제안에 현재/제안 분류, 근거 요약, 구체적 절세 금액이 함께 표시된다.
- [ ] 사용자가 "접대비 유지"를 선택하면 같은 거래·패턴에 재질문하지 않는다.
- [ ] 사용자 확정 전까지 canonical VAT fact·매입세액공제 계산이 바뀌지 않는다.
- [ ] AI가 계정과목을 자동으로 수정하지 않는다.
- [ ] UI에서 "이미 확정된 불공제 설명"과 "재분류 제안"이 서로 다른 자리에 노출된다.
- [ ] 절세 후보가 0건이면 관련 UI 요소가 아예 나타나지 않는다.
- [ ] tenant·사업장·기간 격리와 PII 최소화를 유지한다.
- [ ] 대표 fixture(양성/음성 근거 각각)와 브라우저 E2E로 오탐 방지를 검증한다.

## 11. Out of Scope

- 접대비 외 법정 불공제 사유(③⑤⑥⑦⑧)의 재분류 탐지
- 부가세 외 세목의 절세 가능성 탐지
- 재분류 자동 확정, 기장 계정과목 자동 수정
- 원래 불공제 판정 자체의 정당성을 포괄 재검증하는 감사 기능
- 세무사 자문 대체, 법적 책임 이전
